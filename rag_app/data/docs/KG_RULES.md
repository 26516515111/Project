# 知识图谱构建架构与设计原则 (KG Construction Guide)

## 1. 宏观领域顶层架构 (Upper Ontology)
本项目的知识图谱设计目标为涵盖三大类船舶核心系统：
1. **船舶发动机 (Ship Engines)**
2. **电气设备 (Electrical Equipment)**
3. **液压系统 (Hydraulic Systems)**

**架构原则：**
- **顶层骨架人工维护**：三大类的分类节点、该分类下极其粗粒度的子系统目录等，未来将采用“代码级硬编码”或“静态人工配置文件”的方式进行维护和合并，不由大模型自动抽取（防止发生层级断裂或循环依赖）。
- **底层细节模型抽取**：语言模型（LLM）的唯一职责是：针对当前提供的一份特定说明书（例如：某个具体的电气设备），抽取出该设备的具体组件、故障现象、故障原因和排查方法。

## 2. “防缠绕”的消歧与切块抽取机制
针对大模型自动融合后极易产生的“毛线球”问题（例如：A系统和B系统分别抽出一个完全同名的“电源模块”泛化节点，导致被错误合并进而使A/B系统发生逻辑连线缠绕），本流水线制定如下代码与提示词级别约束：

1. **带上下文切块 (Context-aware Chunking)**
   在切分文档阶段 (`kg_pipeline/steps.py` 和 `rag/chunking.py`)：严密提取该段落所处的完整目录层级 (`heading_path`，如 `[H1] 主机说明 -> [H2] 供电模块`)。
   将此完整信息作为先验上下文，投喂给LLM！

2. **带前缀的强制具象化命名 (Prefix-based Entity Disambiguation)**
   LLM 在抽取时，被严格要求：绝不能单独抽取类似“指示灯不亮”、“过滤器”这类脱离上下文的词！必须结合它所在系统/章节的具体含义，对该实体进行前缀补全。
   - 错误范例：`接触器`、`通讯故障`
   - 正确范例：`GCWJ-01主板_接触器`、`VDR接口_通讯故障`

3. **收敛的关系骨架**
   一切关系都被限制为以 `Equipment`(设备整体) 或 `Component`(具体零部件) 为核心，只允许：
   - `Equipment` HAS_COMPONENT `Component`
   - `Equipment/Component` HAS_FAULT `FaultPhenomenon` (故障现象)
   - `FaultPhenomenon` CAUSED_BY `FaultCause` (故障原因)
   - `Equipment/FaultPhenomenon` REPAIRED_BY `RepairMethod`

## 3. 面向未来的扩展
此流水线不仅适用于当前的电气设备类文件。当引入“船舶发动机”相关语料时：
- 模型无需知道如何构建整个树状大纲，仅需依循同样的Prompt要求，在抽取“活塞环磨损”时标明属于具体哪款“二冲程主机”即可。
- 最终图谱在合并脚本（或者直接通过手工图谱写入）的帮助下，将被自动归档在预设好的上层本体(Ontology)之下，从而形成逻辑极其清晰、无横向污染的星系图谱。
